# 公众号浏览器发布 · 登录态「扫码授权 + 可配有效期」落地设计文档

> 版本:v1.0(架构综合稿) · 基线代码:`backend/` 现状(2026-07)
> 目标读者:后端开发 / 运维 / 教学验收
> 唯一事实源约定:登录态有效期以 **DB 时间戳** 为准,storage_state 磁盘文件为凭据本体;二者以 `app_id` 关联。

---

## 1. 概述与目标

### 1.1 背景(现状事实)
真实发布走浏览器自动化(`BrowserChannel.publish`),整条链复用磁盘上的 `secrets/wx_state/{app_id}.json`(storage_state)作为登录态,**全仓无任何有效期记录**:

- 存:`backend/scripts/wx_login.py` 有头扫码后 `ctx.storage_state(path=...)` 落盘。
- 读:`BrowserChannel._state_path(app_id)` 仅判 `p.exists()`。
- 运行时唯一失效判定:`browser_channel.py` LOGIN 段 `re.search("token=(\d+)", page.url)` 取不到即 `raise WxApiError(-1, "登录态已失效…")`。

**当前最大隐患(已核对 `errors.py`):** `errcode=-1` 在 `WX_ERROR_TABLE` 中归类为 `RETRYABLE`。于是「登录态过期」被当成「系统繁忙」,经 `executor._fail` 进指数退避、`_retry_due` 每分钟盲目重试,直至 `retry_count>=max_retry` 落入 `_dead_letter_alerts` 死信 —— 既淹没告警,又无法区分「需人工扫码」与「内容/选择器故障」。

### 1.2 目标
1. 给登录态引入 **可配有效期(默认 2 天 / 48h,行级可覆盖)**,发布前置校验,过期号根本不进发布队列。
2. 「登录态过期 / 被踢」建成 **一等状态**(独立枚举,不污染 `TaskStatus`),用 **专用 errcode** 短路重试。
3. 续扫做成 **可无人值守的运维闭环**:临期自动告警 → 管理员扫码 → 落盘回写 → 窗口内全自动发布。
4. 过期期间任务 **挂起等授权(留 SCHEDULED)**,而非失败重试 / 死信;续扫后下一 tick 自然放行,天然幂等(`biz_key` 去重)。

### 1.3 设计取舍(三份草案的收敛)
| 争议点 | 草案分歧 | 本文决策 | 理由 |
|---|---|---|---|
| 元数据落点 | A:独立表 `mp_wx_auth`;B/C:`mp_account` 增列 | **`mp_account` 增列** | 登录态与账号 1:1、`account_type∈{1,2}` 才有意义但空列成本极低;复用现有 `last_verified_at` 模式,不新增 join,不新增 18 表之外的表,教学数据字典改动最小。审计另建轻量事件表(见 4.3)。 |
| 状态枚举 | 三家一致:独立 `WxLoginStatus` | **采纳** | 绝不污染 `TaskStatus` / `mp_account.status`(1/2/0 语义正交)。 |
| 过期任务处置 | 三家一致:挂起 SCHEDULED,不进死信 | **采纳,但校验位置前移到 tick 查询层** | 见 6.2:在 `_publish_due`/`_retry_due` 的 SQL 里 join 跳过过期号,任务从不进入 `execute_publish`,状态零扰动。 |
| 续扫 QR 呈现 | A:CLI/后台二选一;B:实时 QR 服务;C:CLI | **一期 CLI 保底 + 后台入口占位** | 生产无头 + 实时 QR 时效(几十秒)风险高,详见 5.2 与开放问题。 |

## 2. 数据模型

### 2.1 `mp_account` 增列(复用 `last_verified_at` 模式)
DDL 落 `deploy/mysql/init/01_schema.sql`(紧跟 `last_verified_at`),模型落 `backend/app/models/mp_account.py`。

```sql
-- mp_account 表内新增(紧跟 last_verified_at):
wx_login_captured_at DATETIME     NULL COMMENT '最近一次扫码成功、storage_state 落盘的时刻(UTC),仅由续扫回写',
wx_login_expires_at  DATETIME     NULL COMMENT '= captured_at + ttl_hours;发布前置与巡检唯一时间判据',
wx_login_ttl_hours   SMALLINT UNSIGNED NOT NULL DEFAULT 48 COMMENT '行级有效期(小时),覆盖全局 WX_LOGIN_TTL_HOURS;微信真实窗口未定死时按号微调',
wx_login_status      VARCHAR(16)  NOT NULL DEFAULT 'UNAUTHORIZED' COMMENT 'WxLoginStatus 枚举快照',
wx_login_alerted_at  DATETIME     NULL COMMENT '续扫告警去重戳:告警后置时间,续扫成功清空',
-- 索引:
KEY idx_login_status (wx_login_status)
```

SQLAlchemy 侧(`MpAccount` 类内,类型对齐):
```python
wx_login_captured_at: Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
wx_login_expires_at:  Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
wx_login_ttl_hours:   Mapped[int]  = mapped_column(SmallInteger, nullable=False, default=48)
wx_login_status:      Mapped[str]  = mapped_column(String(16), nullable=False, default="UNAUTHORIZED")
wx_login_alerted_at:  Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
```

**关键决策:`expires_at` 落库时由 `captured_at + ttl_hours` 一次算死**,后续所有判定只比 `now < wx_login_expires_at` 单字段,不再读配置、不看文件 mtime —— 零歧义,配置漂移不影响已授权号。行级 `wx_login_ttl_hours` 优先于全局配置。

### 2.2 可配窗口(config.py)
`backend/app/core/config.py` 的 `Settings` 新增:
```python
WX_LOGIN_TTL_HOURS: int = 48    # 全局默认有效期=2天(硬要求:必须可配,非硬编码常量)
WX_LOGIN_WARN_HOURS: int = 6    # 临期提前量:进入 EXPIRING 并触发续扫告警的提前小时数
```
`WX_LOGIN_TTL_HOURS` 是「2 天可配窗口」的落点:微信真实窗口一旦证实,只改这一处(或按号改 `wx_login_ttl_hours`)。

### 2.3 与 storage_state 文件的分层
- **文件即凭据**:`secrets/wx_state/{app_id}.json` 仍是登录态本体(`.gitignore` 已忽略),DB 不存 cookie/密文。
- **DB 即台账**:只存有效期元数据 + 状态。二者以 `app_id` 关联,`captured_at` 由落盘那一刻决定,与文件写盘严格同刻,杜绝「文件换了但 DB 时间没更新」的漂移。
- **一致性顺序**:先原子落盘文件(`.tmp` → rename),再写 DB;写库失败可靠文件 + 下一 tick 探测补正(见 7)。

### 2.4 `publish_task` 不新增列
复用现有 `last_errcode/last_errmsg/next_retry_at/retry_count/dispatched`。授权过期时 `last_errcode` 写专用码 `WX_AUTH_EXPIRED`,`tick` 与 `executor` 靠它与真失败区分。

## 3. 授权状态机

### 3.1 新增枚举(`backend/app/core/states.py`)
```python
class WxLoginStatus(StrEnum):
    """mp_account.wx_login_status —— 登录态授权状态,独立于 TaskStatus。"""
    UNAUTHORIZED = "UNAUTHORIZED"  # 从未扫码 / storage_state 文件缺失(初始态)
    AUTHORIZED   = "AUTHORIZED"    # 扫码成功且 now < expires_at,可全自动发布
    EXPIRING     = "EXPIRING"      # now >= expires_at - WARN_HOURS 且仍 < expires_at(仍可发,已告警续扫)
    EXPIRED      = "EXPIRED"       # now >= expires_at,或运行时 token 探测失败(到期)
    REVOKED      = "REVOKED"       # 被微信主动踢下线/异地登录/管理员手动置位(时间未到但 token 失败)
```

### 3.2 迁移白名单(纳入现有 `ensure_transition` 机制)
```python
WX_LOGIN_TRANSITIONS: dict[str, set[str]] = {
    WxLoginStatus.UNAUTHORIZED: {WxLoginStatus.AUTHORIZED},
    WxLoginStatus.AUTHORIZED:   {WxLoginStatus.EXPIRING, WxLoginStatus.EXPIRED,
                                 WxLoginStatus.REVOKED, WxLoginStatus.AUTHORIZED},  # 自环=续扫刷新
    WxLoginStatus.EXPIRING:     {WxLoginStatus.AUTHORIZED, WxLoginStatus.EXPIRED, WxLoginStatus.REVOKED},
    WxLoginStatus.EXPIRED:      {WxLoginStatus.AUTHORIZED},   # 重扫复活
    WxLoginStatus.REVOKED:      {WxLoginStatus.AUTHORIZED},   # 重扫复活
}
# 注册:TABLE_TRANSITIONS["mp_account_login"] = WX_LOGIN_TRANSITIONS
```
> 注:`TABLE_TRANSITIONS` 的 key 是逻辑段名而非物理表名(现有 `collect_article/content_article/publish_task` 已是此风格),故用 `"mp_account_login"` 与 `mp_account` 档案的 `status` 段解耦。

### 3.3 三个驱动源(无歧义)
1. **时间驱动**(tick 巡检):按 `expires_at` 与 `now` 比较,推 `AUTHORIZED → EXPIRING → EXPIRED`。
2. **事件驱动·续扫成功**(`wx_login.py` 回写):置 `AUTHORIZED`(自环刷新 `expires_at`)。
3. **事件驱动·运行时失效**(`BrowserChannel` LOGIN 段 token 失败):置 `EXPIRED`(时间已过)或 `REVOKED`(时间未到但被踢)。

`AUTHORIZED → AUTHORIZED` 自环用于续扫刷新,幂等。所有非法迁移抛 `IllegalTransition`。**授权状态与 `TaskStatus` 正交:授权过期绝不把任务打成终态 FAILED。**

## 4. 续扫运维流程

### 4.1 谁触发、何时触发
不靠人盯。`scheduler.tick` 新增 `_login_watch` 段(见 6.1),每分钟巡检所有真实号(`account_type∈{1,2}`):
- `now >= expires_at - WARN_HOURS` → 推 `EXPIRING`;
- `now >= expires_at` → 推 `EXPIRED`;
- 对**首次**进入 EXPIRING/EXPIRED 的号(幂等,靠 `wx_login_alerted_at` 是否为空去重,复刻 `_dead_letter_alerts` 的「未告警过才发」模式)推送续扫工单,置 `wx_login_alerted_at=now`。

「何时」= 到期前 `WARN_HOURS`(默认 6h)给管理员留扫码时间,而非等发布失败才被动发现。

### 4.2 QR 呈现给管理员(一期 CLI 保底,后台入口占位)
微信登录 QR 必须实时从后台取(有几十秒时效,不能预生成)。分层落地:

- **一期(保底,可无人值守到「告警级」)**:告警工单经 `notify()`(→ `ALERT_WEBHOOK_URL` 企业微信机器人)推送,内容含 `mp_name / app_id / expires_at 倒计时 / 「请管理员在部署机执行 python scripts/wx_login.py {app_id} 扫码续期」`。管理员在部署机(有头 Chromium,`headless=False`)用**该号管理员微信**扫码。
- **二期(后台续扫入口,占位)**:参照 `mp_manager.service.verify_account`(账号生命周期现成入口)在 `mp_manager` 挂 `POST /mp/{id}/wx-reauth`,服务端起有头会话把实时 QR 截图/透传前端;前端仪表盘按 `wx_login_status` 分组展示各号与 `expires_at` 倒计时,运维可主动点任意号发起续扫。二维码可临时写现有 `mp_account.qrcode_url` 列或内存,不入状态机。
  - **安全边界(必补)**:续扫入口须校验发起人对该号有发布权(`MpAccountAssign.perm_level >= 4`),否则任意人可触发扫码劫持登录态。

### 4.3 扫码成功如何持久化信任(让 headless 复用)
改造 `backend/scripts/wx_login.py` 的 `capture(app_id)`:把「落盘 + 回写 DB」抽成可复用函数 `persist_login(app_id, out_path)`,在 `ctx.storage_state(path=...)` 落盘成功后调用,在**一个事务**里:
```
UPDATE mp_account SET
  wx_login_captured_at = utcnow(),
  wx_login_expires_at  = utcnow() + (wx_login_ttl_hours 或 WX_LOGIN_TTL_HOURS) 小时,
  wx_login_status      = 'AUTHORIZED'   (经 ensure_transition 从任意态迁入),
  wx_login_alerted_at  = NULL
WHERE app_id = :app_id;
```
并写一条轻量审计事件(建议新建 `wx_login_event` 表:`app_id / event(CAPTURE/EXPIRE/REVOKE/REAUTH_PUSHED) / operator / created_at`,记录谁在何时续的扫,便于运维追责)。

**信任前提(修正草案缺陷)**:现状 `wx_login.py` 即使 URL 未见 `token=` 仍落盘 cookies,可能落一个「半有效」登录态使 `expires_at` 虚高。**本设计要求:仅当 `token=` 出现(真正进入后台)才写 `AUTHORIZED` 并计算 `expires_at`;未见 token 时保存文件但 `wx_login_status` 维持原值 / `UNAUTHORIZED`,不刷新有效期。**

## 5. 窗口内全自动发布

### 5.1 单出口前置校验(核心拦截)
在 `WxGateway.publish` 真实分支(`gateway.py`,系统唯一微信出口)开头,`is_mock=False` 时先查该 `mp` 的登录态:
```python
# 伪码
if not self.is_mock(mp):
    if mp.wx_login_status in (EXPIRED, REVOKED, UNAUTHORIZED) \
       or (mp.wx_login_expires_at and utcnow() >= mp.wx_login_expires_at):
        raise WxApiError(WX_AUTH_EXPIRED, f"{mp.mp_name} 登录态已过期,需重新扫码授权")
    return await BrowserChannel().publish(mp.app_id, articles, publish_type)
```
`AUTHORIZED` 与 `EXPIRING` 均放行(EXPIRING 仍在窗口内,只是并行触发了续扫提醒),保证临期不中断在途发布 —— 这正是「扫码后 2 天窗口内全自动」的落点。过期则**不启浏览器**直接抛错。

> `gateway.publish` 是 async 且当前不持有 `db`。落地方式:由 `executor.execute_publish`(持 `db` 与 `mp` 对象)在调 `gateway.publish` **前**做同一校验(mp 对象已含新列,无需额外查询),`gateway.publish` 内保留一道兜底。这样避免给 `WxGateway` 注入 db 依赖。见 6.2 与开放问题。

### 5.2 确认仍在窗口内(双保险)
- **静态判定(发布前)**:读 `wx_login_expires_at`,`now < expires_at` 即在窗口。
- **动态兜底(发布中)**:`BrowserChannel` LOGIN 段保留 token 探测。即便 DB 说没过期但微信实际已踢下线,token 缺失 → 抛专用码 `WX_AUTH_EXPIRED`(**由现状的 `-1` 升级为专用码**,这是修复 RETRYABLE 误分类的关键)→ executor 特判后降级该号 EXPIRED/REVOKED、写状态并触发续扫。两道判定分别覆盖「时间到了」与「时间没到但被提前失效」。

### 5.3 发表框自动关群发通知 + 点发表(沿用现状,不改)
`browser_channel.py` 的 `confirm_labels` 逐个放行序列(`同意以上声明 / 无需声明并发表 / 继续发表 / 确认发表 / 确定`,最多 6 次循环,检出 `published_link` 即停)**本身就是「自动点发表 + 放行连弹声明」的实现**。

要点:窗口内登录态可信,`publish_type==1`(发表到主页,不推粉丝)点完 `button.mass_send` 后固定标签表即可走通到 `published_link`,全程无人。`publish_type==2`(群发)走 `_wait_result(180s)` 等管理员扫码/短信的**二次人工确认** —— 这是群发环节微信另一次风控确认,**与本「登录态 2 天有效期」是不同环节,不纳入 `WxLoginStatus` 状态机**。本设计只保证「进入发表流程前登录态一定有效」。

> 待补(需真机 inspect):发表框「群发通知」开关的选择器,让 `publish_type==1` 自动关开关再点发表。当前该开关由人工关闭。

## 6. 失败兜底(检测 → 暂停 → 告警 → 重试而非死信 → 幂等)

### 6.1 专用错误码(`backend/app/modules/wx_gateway/errors.py`)
新增 `AUTH_EXPIRED` 归类常量与错误码进 `WX_ERROR_TABLE`:
```python
AUTH_EXPIRED = "auth_expired"     # 新归类:暂停该号队列,不退避,不死信,告警续扫
WX_AUTH_EXPIRED = 48101           # 新错误码
WX_ERROR_TABLE[48101] = ("登录态过期需重新扫码授权", AUTH_EXPIRED, "暂停该号队列;不计退避;告警续扫")
```
`WxApiError` 增便捷属性 `is_auth_expired`(`self.category == AUTH_EXPIRED`)。
**必须修复**:`browser_channel.py` LOGIN 段 `WxApiError(-1, ...)` 改为 `WxApiError(WX_AUTH_EXPIRED, ...)` —— 否则登录态过期仍被 `-1` 的 RETRYABLE 语义盲目重试。

### 6.2 检测与「暂停该号队列」
**检测两处**:(a) 前置时间校验(5.1);(b) 运行时 LOGIN token 探测失败(5.2)。

**暂停策略(位置前移,状态零扰动)——本文对三份草案的关键收敛**:
- `scheduler._publish_due` 与 `_retry_due` 的查询 **join `mp_account`,过滤掉 `wx_login_status ∈ {EXPIRED, REVOKED, UNAUTHORIZED}` 或 `wx_login_expires_at <= now` 的号**。过期号的任务**根本不进 `execute_publish`**,保持 SCHEDULED 原地挂起,不迁 FAILED、不计 `retry_count`、不触发死信。
  > 为何前移:`execute_publish` 一进门就 `SCHEDULED → PUBLISHING` 再调 `gateway.publish`。若在其内部才拦截,任务已在 PUBLISHING,需额外回滚。查询层过滤最干净。
- **运行时突发失效**(时间没到但被踢,已进 `execute_publish`):`gateway.publish` 抛 `WX_AUTH_EXPIRED` → executor 特判 `e.is_auth_expired`:①把 `mp.wx_login_status` 降级 EXPIRED/REVOKED、清 `wx_login_alerted_at`;②任务不进死信(不自增 retry_count),回退留 SCHEDULED 等续扫。验收标准:**任务永不进死信**。

### 6.3 告警(号级一次,不淹没死信渠道)
登录态失效告警走 `notify()`(→ `ALERT_WEBHOOK_URL`),内容含 `mp_name/app_id/expires_at/「请管理员扫码续期」`。**收敛到号级一次**:靠 `wx_login_alerted_at` 边沿去重(进入 EXPIRING/EXPIRED 且 `alerted_at` 为空才发),续扫成功清空。与 `_dead_letter_alerts` 的「已告警不重发」模式一致,彻底与内容/选择器死信告警分流。

### 6.4 任务重试而非死信 + 幂等
- 因登录态过期挂起/降级的任务 **不计入 retry_count、不进死信**;续扫后 `wx_login_status → AUTHORIZED`,下一 tick `_publish_due` 正常捞起发布。
- **幂等三重**:①`biz_key=article:{id}` 唯一键(`uk_biz_key`),过期期间不新建重复任务;②任务是「同一条重发」而非新建;③`BrowserChannel` 已有「检出 `published_link` 即停」;发表前可选探测该文是否已有成文链接,已发则短路返回 PUBLISHED,防「过期误判 → 重发 → 重复推粉丝」(`publish_type==2` 群发风险更高,故过期必须尽量前置拦截)。

## 7. 前端呈现(账号授权状态列)

`mp_manager` 账号列表接口(`router.py`/`schemas.py`)输出对象新增只读字段:
- `wx_login_status`(枚举值,前端映射:`AUTHORIZED`=绿「已授权」/`EXPIRING`=黄「临期 · 剩 Xh」/`EXPIRED`/`REVOKED`=红「需续扫」/`UNAUTHORIZED`=灰「未授权」)。
- `wx_login_expires_at`(前端算倒计时)。
- **仅对 `account_type∈{1,2}` 真实号展示该列**;`account_type==3` 模拟号展示「Mock(无需授权)」。
- 每行「需续扫/临期」态旁挂「去续扫」按钮 → 触发 4.2 二期入口(一期先展示 CLI 命令提示)。

## 9. 实施顺序与验收
先做「数据模型 + 状态机 + 专用错误码 + 单出口/查询层校验」(纯后端,可 Mock 单测);再做「tick 巡检 + 告警 + wx_login.py 回写」(运维闭环);最后做「前端列 + 二期后台续扫入口」。验收铁律:**任何登录态过期场景,任务永不进死信、永不重复发文,续扫后自动恢复。**

## 10. 待真机 / 运维确认的开放问题
1. 微信 storage_state 真实有效期是否恰为 2 天、固定窗口还是**滑动窗口**(每次操作续期)?若为滑动,发布成功后刷新 `expires_at`(AUTHORIZED 自环已预留);TTL 先按保守 40–48h 上线再校准。
2. 续扫是否必须真正进入后台(`token=` 出现)才算刷新,还是仅扫码即刷新?真机验证。
3. 运行时区分「到期 EXPIRED」与「被踢 REVOKED」需更强页面特征(登录页/二维码 DOM),需 `--inspect` 校准。
4. **最大部署约束**:publish-runner(headless 发布)与 login-runner(有头扫码)是否**同机、共享 `secrets/wx_state` 目录**?若 Celery worker 与登录机分离,storage_state 文件无法跨机复用,需共享卷/对象存储。**当前浏览器发布跑在本机,不在服务器**——要让服务器全自动发,须先定 runner 拓扑。
5. 生产发布机是否有显示环境?有头 Chromium 扫码需桌面或 xvfb;二期「服务端起有头会话实时透传 QR」在纯 headless 服务器可行性未验证,可能仍需回退部署机 CLI 扫码。
6. `publish_type=1` 常拿不到稳定 `published_article_id`,幂等短路较脆;群发重复会真推粉丝——业务对「过期期间任务延迟发布(等授权)」的可接受度需确认。
